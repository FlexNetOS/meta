# Agent Skills Catalog Source Trace

This note records the current non-destructive source trace for
`.agent/skills-catalog.md`.

## Current generator

The checked-in root catalog is generated by the VS Code extension
`aliabbaschadhar.copilot-superpowers` using its `aiSkills.refreshCatalog`
command, not by `kasetto`.

Local evidence on this workstation:

- Extension install: `.local/share/vscode/extensions/aliabbaschadhar.copilot-superpowers-0.3.2/`
- Bundled index: `.local/share/vscode/extensions/aliabbaschadhar.copilot-superpowers-0.3.2/assets/skills_index.json`
- Remote cache: `/home/drdave/.config/Code/User/globalStorage/aliabbaschadhar.copilot-superpowers/skills_index.json`
- Default remote index URL embedded in the extension:
  `https://raw.githubusercontent.com/sickn33/antigravity-awesome-skills/main/skills_index.json`

The extension initializes from the bundled `assets/skills_index.json`, then
merges the global-storage `skills_index.json` by appending IDs not already in
the bundled index. On this machine that means:

- bundled index: 1,259 skills
- global-storage cache: 1,693 skills
- appended from cache: 434 new skill IDs
- merged catalog: 1,693 skills across 109 categories

A local reproduction of the extension render logic exactly matched the committed
`.agent/skills-catalog.md` bytes for the refresh captured in PR #94.

## Relationship to `vault_hub`

`vault_hub/.agent/skills-catalog.md` is an older copy of the same catalog family
at 1,508 skills across 97 categories. It is evidence of prior catalog lineage,
but it is not the source that produced the current root refresh.

## Relationship to `kasetto`

`kasetto` is still relevant to the agent-environment layer: it syncs skills,
commands, and MCP assets into agent environments from configured sources and
lockfiles. It is not currently the generator for the root
`.agent/skills-catalog.md` file.

## Non-blocking upgrade path

Do not revert the refreshed catalog. Treat it as a generated projection until a
repo-native source of truth is promoted. Future additive options:

1. Add a small checked-in renderer that reproduces the extension catalog output
   from explicit JSON inputs.
2. Store the catalog inputs under a dedicated source path or declare the VS Code
   extension + remote cache as external inputs.
3. Decide whether `kasetto` should become the durable installer/sync layer only,
   or also own catalog rendering through a new non-destructive subcommand.
4. Keep generated outputs separate from source-truth inputs so future refreshes
   are reproducible and reviewable.
